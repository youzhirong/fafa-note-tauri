# 架构说明

## 设计目标

1. **一套代码多端运行**：Web / 桌面 / 移动共用同一前端。
2. **好扩展**：新增功能、替换存储引擎时，改动局限在少数文件，UI 不受影响。
3. **纯前端可用**：无后端也能完整工作（数据存本地，备份走文件）。

## 分层

```
┌─────────────────────────────────────────┐
│  UI 层      layouts / components / views   │  只管展示与交互
├─────────────────────────────────────────┤
│  状态层     stores (Pinia)                 │  notes / settings / ui
├─────────────────────────────────────────┤
│  服务层     services                       │  搜索 / 备份 / HTTP
│             repository (INoteRepository)   │  ← 存储抽象接口
├─────────────────────────────────────────┤
│  存储实现   IndexedDbNoteRepository        │  可替换（SQLite…）
├─────────────────────────────────────────┤
│  平台能力   platform + Tauri 插件           │  文件/对话框/HTTP
└─────────────────────────────────────────┘
```

**依赖方向自上而下**：UI 依赖 store，store 依赖 service/repository 接口，
接口的具体实现可随平台/需求替换。UI 永远不直接碰存储或平台 API。

## 关键设计取舍

### 为什么存储用 IndexedDB 而不是 SQLite？

你的 Web 版托管在静态页面、**无后端**。Tauri 的 `plugin-sql`（SQLite）只能在 Tauri 容器内运行，
纯浏览器跑不了。IndexedDB 浏览器原生、Tauri 的 WebView 同样支持，所以**一套实现跑全平台**。

数据量很大、需要更强搜索时，可在桌面端换成 SQLite（FTS5 全文搜索）——
因为有 `INoteRepository` 接口，换实现不影响上层。见 [扩展指南](扩展指南.md)。

### 为什么搜索用 MiniSearch？

IndexedDB 没有内置全文搜索。MiniSearch 在内存建倒排索引，几千~几万条笔记关键词搜索是毫秒级，
支持前缀、模糊、字段加权。store 在加载/增删改时同步维护索引。

### 平台差异如何隔离？

统一通过 `platform/index.ts` 的 `isTauri()` 判断，差异集中在两处服务：

- `BackupService`：Tauri 读写磁盘目录；Web 用浏览器下载 + 文件选择。
- `HttpService`：Tauri 用 `plugin-http` 直连（无 CORS）；Web 套用户配置的代理地址。

Tauri 插件均用**动态 import**，确保纯 Web 打包时不会因缺少 Tauri 运行时而报错。

## 数据模型

见 `src/types/index.ts`。要点：

- 所有实体带 `id / createdAt / updatedAt`，便于排序与未来同步。
- 备份结构 `BackupBundle` 带 `version` 字段，方便格式升级做兼容。
- 新增字段尽量「只增不改」，并在 Dexie 的 `version(n).stores().upgrade()` 里做迁移。

## 数据流示例（编辑一条笔记）

```
用户在 NoteEditor 打字
  → 本地草稿 ref 变化
  → 防抖 400ms 后调用 notesStore.updateNote()
  → repository.saveNote() 写入 IndexedDB
  → searchService.update() 刷新搜索索引
  → notes 数组更新 → visibleNotes 计算属性刷新 → 列表重渲染
```
